# DESIGN — Content Pipeline

> Living document. Defines how Pathfinder source material (and our
> original content) flows from raw extracts to runtime data.
> Load-bearing for the extract-by-extract import workflow (per
> *Workflow* §3).

---

## 1. Stance

The pipeline is the project's **production line**. Every piece of
content — a spell, a feat, a stat block, an NPC, an AP encounter, an
original quest — passes through it.

Design goals:

1. **Lossless source preservation.** The original extract is preserved
   verbatim alongside its converted form, so we can re-derive when our
   conversion rules change.
2. **Per-extract Q&A capture.** Every conversion decision is logged
   inline with the converted record (per *Workflow*).
3. **Schema-validated runtime.** The runtime never loads a record that
   wasn't validated against the canonical schema.
4. **Diffable in git.** Source files, converted files, and decision
   logs are all human-readable text under version control.
5. **Fast-iteration friendly.** A re-import of one extract should not
   require rebuilding the world.

## 2. Pipeline Stages (proposed; pending T-Q3 + T-Q15)

```
[Raw Source]   →   [Source Tier]   →   [Canonical Tier]   →   [Runtime Tier]
   │                    │                    │                      │
   │                    │                    │                      │
 Paizo PDF,         human-edited         Remaster-shaped       SQLite indexed
 community         per-extract           validated, schemed    + in-memory cache
 dataset,          file in `Content/`    JSON                  + graph projection
 third-party                              in `Content/`         (per T-Q3)
                                          (or build-only)
```

### 2.1 Raw Source

- The original published material (PDF, web page, scrape, or other).
- **Not** committed to the repo (license-agnostic per Q33, but bulky).
- Lives in a dev-machine folder; referenced by URI/path in the source-tier
  file's frontmatter for traceability.

### 2.2 Source Tier (`Content/Source/...`)

- Per-extract files written by hand during the import workflow.
- Format (resolved CP-Q1 / T-Q15): **hybrid** —
  - **Markdown with YAML frontmatter** for prose-heavy content
    (lore, scenes, dialogue, descriptions, AP chapters, NPC bios).
  - **Pure YAML** for stat-heavy content (spells, feats, items,
    monsters, ancestries, classes, runes, encounters).
- Each file:
  - Identifies the source (book, page, edition, year) in
    frontmatter / top YAML.
  - Holds the **converted-to-Remaster** content per user decisions.
  - Includes an **inline `decisions:` YAML block** (resolved
    CP-Q3) recording every Q&A from the extract import session,
    with timestamps and *Open Questions* IDs where applicable.
  - Cross-references related extracts via plugin-scoped IDs
    (`<plugin>:<slug>`).

### 2.3 Canonical Tier (`Content/Build/canonical.sqlite`)

- Generated by the build pipeline from the Source tier.
- Schema-validated.
- Indexed for query (lookup by trait, rank, tradition, etc.).
- Single source of truth for the runtime.

### 2.4 Runtime Tier

- Engine loads canonical-tier records into in-memory game-side
  structs (per T-Q3 hybrid).
- Graph projection built once at boot.
- Save data is a parallel system (see *Save and Persistence*).

## 3. Extract-by-Extract Import Protocol

(This restates *Workflow* §3 with pipeline-specific operational detail.)

For each extract:

1. The user provides the raw source (paste, link, file).
2. The assistant reads it and **asks every conversion question
   raised** (Remaster shape decisions, lore alignment, tagging,
   placement, edge cases).
3. The user answers per-extract.
4. The assistant writes the source-tier file with the converted
   content + decisions log.
5. The build pipeline regenerates the canonical-tier index.
6. (Optional) the assistant runs the validator to confirm.
7. Commit on the working branch with a message summarizing the
   extract.

## 6. Open Questions (Content Pipeline-Local)

### Format and tooling

- ~~**CP-Q1**~~ **Resolved (mirrors T-Q15).** Option (f) hybrid:
  **Markdown-with-YAML-frontmatter for prose-heavy content (lore,
  scenes, dialogue, descriptions); pure YAML for stat-heavy content
  (spells, feats, items, monsters, classes, ancestries,
  encounters).** Single canonical schema reachable from either. See
  *Tech and Engine* T-Q15 resolution for rationale.
- **CP-Q2** Validation gate model. Options:
  - **(a) JSON Schema** — well-known, tool support.
  - **(b) CUE** — typed, composable, expressive.
  - **(c) Pydantic / typed Python models** — code-as-schema.
  - **(d) Custom validator written in C# tied to the runtime types.**
- ~~**CP-Q3**~~ **Resolved.** Decisions block as **inline YAML
  inside the source-tier file** (Option (a)). Every extract carries
  its own decision log with the converted content; no sidecars.
- **CP-Q4** Build-tool language. Options:
  - **(a) Python** — best ecosystem for parsing/regex/scraping.
  - **(b) C# (matching engine choice)** — single-language stack.
  - **(c) Rust** — fast, strict, more friction.

### Source acquisition

- ~~**CP-Q5**~~ **Resolved.** **Paste in chat for small extracts;
  file drop in `Content/Source/_raw/` for large ones.** URL fetch
  available when the user explicitly hands me a URL. The
  `_raw/` folder is gitignored (raw sources are dev-only, not
  committed per §2.1).
- **CP-Q6** OCR pipeline if PDFs only (Tesseract, cloud OCR, manual
  cleanup)?
- **CP-Q7** Scrape policy for community datasets (Archives of Nethys,
  PF2 Easytool, Foundry pf2e package): pull live, snapshot, or import
  per-extract by hand?
- **CP-Q8** Image asset pipeline for stat-block art, region maps,
  iconic portraits — separate folder, separate commit policy?

### Conversion rules

- **CP-Q9** When PF1e content has no Remaster equivalent, default
  policy: skip / approximate / hand-author?
- **CP-Q10** When community datasets disagree (AoN vs Foundry vs
  errata), which wins by default? (Or: always ask user per case?)
- **CP-Q11** Errata application — track or fold-in?
- **CP-Q12** Original (non-Paizo) content tagging — flag explicitly
  in source-tier files for filtering?
- **CP-Q13** Cross-extract identifier strategy — UUIDs, slugs,
  canonical-name-based, hierarchical?
- **CP-Q14** Localization fields — present in schema from day one
  even if English-only target?
- **CP-Q15** Versioning — does each source-tier file carry a `version`
  field for re-import history?

### Workflow integration

- **CP-Q16** Re-import policy when an upstream errata changes a record
  we already imported (mirrors W-Q6).
- **CP-Q17** Bulk-import authorization — under what conditions, if
  any, can the assistant batch-convert multiple extracts at once
  (currently *Workflow* says forbidden by default; carve-outs?).
- **CP-Q18** Preview / dry-run — does the user see the converted
  output before commit, every time?
- **CP-Q19** Conflict resolution — when an import affects a
  previously-imported record (e.g., new feat references an existing
  spell whose tags need updating), how do we resolve?
- **CP-Q20** Naming conflicts (two sources have the same name for
  different things) — disambiguator policy?

### Runtime loading

- **CP-Q21** Hot-reload during dev — does the engine watch the
  canonical SQLite for changes and live-reload?
- **CP-Q22** Plugin discovery order (per T-Q5) — alphabetical,
  manifest-declared, dependency-graph?
- **CP-Q23** Missing-reference policy — strict (fail boot) or
  permissive (warn and skip)?
- **CP-Q24** Schema migration when we change the canonical schema
  shape — auto-migrate the build, or require source-tier edits?

### Quality and review

- **CP-Q25** Spot-check process — how does the user review converted
  content for fidelity periodically?
- **CP-Q26** Test coverage — auto-tests on conversion (e.g., dice
  expressions parse, references resolve, stats sum correctly)?
- **CP-Q27** AI-assisted conversion — at what stages can dev-time AI
  draft the conversion before user approval (this turn for Q&A,
